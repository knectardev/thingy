The "Fuzzy Repo Routing" Cursor Prompt
Context: I want to enhance the #feature hashtag logic to dynamically route issues to the correct GitHub repository based on a second "context" hashtag.

Goal: When a user submits a message like "Photo upload #feature #thingy", the system should:

Identify the primary token (#feature) and the context token (#thingy).

Search my GitHub account for a repository that best matches "thingy".

Create the issue in that specific repo.

Task List:

Install Dependency: Add fuse.js for lightweight fuzzy searching.

Update Parser: Enhance lib/parser.ts to extract secondary "context" tokens when #feature is present.

Enhance GitHub Handler:

Update lib/handlers/github.ts to include a findBestRepo(searchQuery: string) function.

This function should fetch the list of repositories for the authenticated user (using octokit.repos.listForAuthenticatedUser).

Use fuse.js to find the closest match between the "context" token and the available repo names.

Logic & Error Handling:

If a match is found (with a high confidence score), proceed to create the issue.

If no match is found (or the score is too low), return a specific error message.

UI Update: Update the ActivityFeed component to display "No matching repo found" if the dynamic routing fails, so the "Glass Box" remains transparent.

Logging: Log the "closest match" result and the confidence score for debugging.

Constraint: Ensure the GitHub API call for the repo list is cached or handled efficiently to avoid hitting rate limits on every single capture.

Why this approach works:
Fuse.js: This is the industry standard for what you're asking. It’s tiny, doesn't require a backend database, and handles things like "thingie" vs "thingy" perfectly using Bitap algorithms.

The "Context" Token: By treating the second hashtag as the target, you keep your input syntax clean. You don't have to tell the system "send this to GitHub"; the presence of #feature implies GitHub, and #thingy implies the destination.

Dynamic Scaling: You no longer have to update your code when you start a new project. As soon as you create a repo on GitHub, "Thingy" will be able to find it via the fuzzy search.

A Small Warning on Performance
Fetching the full list of your repos on every capture might add ~500ms of latency. Since you're running on Vercel, I'd suggest having the handler fetch the list once and perhaps store it in a simple global variable (which stays warm between some Lambda executions) or just accept the tiny delay since this is a "personal" tool and not a high-traffic API.
